# 03｜用 Review 控制 AI 编程的粒度

上篇我们写了怎么把项目骨架立起来。

接下来，我同样不会写开发文档之类的东西。

会不会担心实现有偏差？不用担心，一定会有。

但是我说过，软件开发就是构建一个系统，一步步逼近目标。在目标实现之前，所有做的事情跟目标一定都有差距。但大体方向是一致的，这是一个粗糙推进的过程。

举个例子，我们最终的使用方式会长这样。用一个 hello world 说明。

Nginx 配置文件：

```nginx
events {}

http {
    server {
        listen 8080;

        location /hello {
            lua_content app/hello.lua;
        }
    }
}
```

Lua 代码文件：

```lua
return {
    handler = function(r)
        return Response.new("hello world")
    end
}
```

Nginx 指令 `lua_content` 指定了 Lua 代码文件，Lua 代码里已经可以使用 `Response` 这样的对象。

但是第一步我们会做成什么样子呢？就是类似这样。

```nginx
events {}

http {
    server {
        listen 8080;

        location /hello {
            lua_content 200, "hello world";
        }
    }
}
```

这不是我设计的，是 AI 设计的。我也没有跟 AI 说，你这样不符合目标。

这里非常典型地体现了我怎么使用 AI。我只关心每一步是不是往目标逼近，不关心差距有多少，但我知道差距在缩小。

因为这步做完以后，Nginx 已经用上了 Lua 引擎，而且我们还有测试用例。说明不止集成进来了，还集成得正确。

所以这就是用 AI 开发的粒度。每步完成一件独立的事情，并且你能将它讲清楚。它给 Review 的人的感觉应该是清晰、简洁的。

那有没有一个比较好的粒度衡量方式呢？就是 Review。

我在 Nginx 工作多年，Review 工作可以说和开发工作一样重要。我对 Review 的理解就是：这个功能独立有价值，代码简洁清晰。

所以我们可以按 Review 的角度来衡量这个粒度。

我第一步做的，就是让 AI 将 Lua 引擎集成进来。它会自己设计 `lua_content` 的指令，那个是次要的。里面的 Lua 代码只是让它运行，不影响请求和响应。

接着第二步，我让它用文件的方式来管理代码，同样不影响请求和响应。

你看，是不是每步都独立，而且必须有价值。

如果是人写代码，还真像愚公移山。但是 AI 的执行力是真强，这种每个功能都花不了半个小时。我也有 Review 代码，但我只要觉得在设计上没有出现完全不合理的地方，就通过了。

因为我知道后面还要继续 Review，甚至改设计。

到这里完成的情况是：可以用 Lua 引擎，并且已经可以用文件的方式管理代码了。但是使用方式和我们想要的还八竿子打不着。

这篇主要想展示做的事情和用 AI 开发的方式。我们一直以代码驱动，完全没有开发文档。但别忘了，测试和使用文档一直保持和代码同步。
