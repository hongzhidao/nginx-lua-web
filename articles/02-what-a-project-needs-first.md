# 02｜一个项目需要先立住什么

我用 AI 编程的一个核心原则是：先做最小、核心、可验证的东西。

软件开发本质上是在对抗复杂性。编程不是一次性把目标完整实现出来，而是在构建一个系统，并且一步步逼近目标。

所以不管是目标设计，还是实现过程，我都会先收敛到一个最小的核心闭环，并且保证它可以被验证。后面的打磨和扩展，都围绕这个闭环继续推进。

## 最小、核心、可验证

放到一个新项目里，我首先考虑三件事：

- 最小
- 核心
- 可验证

最小，是不急着把东西做全。

核心，是先抓住系统最重要的方向。

可验证，是每一步都有办法判断当前状态是否成立。

这也符合目前 AI 最擅长的事情。它很适合做清晰、明确、局部的任务。

一个项目应该先有一个很小的形态。这个形态不一定好用，也不一定完整，但它必须表达项目的方向，并且可以被验证。只要这个形态成立，后面就可以围绕它继续演进。

## 项目一开始需要什么

基于这个标准，我认为一个项目一开始的最小集合是三类东西：代码、测试、使用文档。

落到目录上，通常会是这几个入口：

```text
src/
tests/
docs/
README.md
```

`src/` 表达项目要做什么。

`tests/` 保证项目当前的状态是可验证的。

`docs/` 放更具体的使用文档，比如 API、示例和更完整的用法说明。

`README.md` 是项目入口，说明项目是什么、怎么开始使用、现在做到哪里、接下来往哪里走。

这三类东西建立起来以后，项目才算真正站住了。因为它不再只是一段代码，而是一个可以被理解、被运行、被检查、被继续推进的系统。

## 文档不是越多越好

这里说的文档，是使用文档，不是开发文档。

不是指 `AGENT.md`、`SPEC.md` 这类给 AI 看的文档。我不排斥它们的价值，在一些项目里它们也确实有用。

但在这个项目里，我更想用代码驱动的方式推进。

因为我相信代码本身就是一种设计。

## AI 擅长什么

AI 擅长清晰明确的局部事情。

因为目前对复杂性的处理，它还有距离。

如果我们帮它缩小范围，让事情明确，它的执行力就会很高。

为什么先做最小？举个例子。

比如我们会实现 `Headers`，它有 `new`、`get`、`set`、`delete` 等方法。

但对用户来说，最核心、最常用的可能就是 `new` 和 `get`。那我就会先实现这两个方法，并且让测试通过。

到下一步，目标就明确变成补充其它功能。其它功能不用再管这个对象的内部结构，基本就是填充功能。

这就符合我对软件开发的理解：开发是逐步逼近目标，而不是一下子实现目标。

## 这个项目的第一步

所以 `nginx-lua-web` 一开始做的事情也很简单。

它不是先做路由，也不是先做完整的 Fetch API，更不是一开始就追求 Hono 那样的开发体验。

虽然这个项目的目标，是让 Nginx 可以用 Lua 写类似 Hono 那样的 Web 应用，但第一步我并没有让 Lua 真正执行应用代码。

我只是先让 Nginx 有一个入口。

这个入口就是 `lua_content`。

访问时一律返回 `404` 响应。

这一步里，源码、测试和文档做的是同一件事。

源码负责建立最小入口：

- 注册 `lua_content` 指令。
- 把它安装成 Nginx location 的 content handler。
- 先返回 `404`，证明请求已经进入这个模块。

测试负责验证这个入口真的存在：

- 检查模块是否声明为 Nginx HTTP module。
- 检查 `lua_content` 指令是否注册。
- 检查 handler 是否被安装到 location。
- 检查当前行为是否就是返回 `404`。

文档负责如实描述当前状态：

- 这个项目要做什么。
- 当前已经实现什么。
- 当前还没有实现什么。
- 怎么构建，怎么测试。
- 下一步准备往哪里走。

这就是我想要的第一步。

不是把功能做完整，而是让源码、测试、文档同时指向同一个最小事实：这个项目已经有了一个 Nginx 入口。

## 同步不是额外规则

测试和文档始终应该和源码同步。

我们以前经常会写类似 `AGENT.md` 的文件，专门强调这类规则：改了代码，要改测试；行为变了，要改文档。

我相信大模型一直在进步。AI 的趋势应该是降低复杂性，不管是开发，还是决策。

目前至少上面这种规则，你不用跟它强调。它会自动遵守得很好。

所以我推荐多实践，多用最新、最强大的大模型。

## 目前完成了什么

到这里为止，这个项目完成了两件事：

- 建立了项目骨架。
- 让这个模块在 Nginx 里跑起来。

这两件事都很小，但它们已经足够作为起点。

项目有了源码，有了测试，有了使用文档。Nginx 也有了 `lua_content` 这个入口。虽然它现在只返回 `404`，但这说明请求已经能进入这个模块。
